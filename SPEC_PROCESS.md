> 记录与 Superpowers 协作生成 SPEC 和 PLAN 的过程，包括 brainstorming 关键节点、冷启动验证、以及规约修订。

## 一、Brainstorming 过程记录

### 1.1 初始状态

- **工作目录**：`D:\zuoye1`（空白项目，无 Git 仓库，无任何文件）
- **开发工具**：Claude Code（Anthropic）+ Superpowers 技能框架
- **项目类型**：A · Coding Agent Harness
- **已知约束**：必须使用 Superpowers 七步工作流、TDD 强制、凭据安全存储、公开仓库 + CI

### 1.2 关键决策节点

#### 决策 1：技术栈选择

智能体追问语言偏好，用户选择 **Java 17**。

- AI 建议：TypeScript（生态最丰富）、Python（AI 生态最强）、Go（分发最方便）、Rust（性能最强）
- 用户选择：Java
- **分析**：Java 的强类型系统 + JUnit/Mockito 测试生态与项目"机制必须是代码"的硬要求高度契合。用户可能对 Java 最熟悉，选择务实。

#### 决策 2：重点维度选择

智能体追问六个维度中哪个做深，用户选择 **记忆（Memory）**。

- AI 分析：治理护栏最容易用代码体现深度；反馈闭环在 coding 场景最自然；记忆需要自己实现存储与检索，挑战最大也最能体现原创工程能力。
- 用户选择：记忆
- **分析**：记忆是 AI 陪伴应用最核心的竞争力，选择它意味着 harness 不仅是一个代码生成器，还封装了领域最佳实践。

#### 决策 3：架构风格

智能体提出三种方案：Spring Boot（偏重）、纯 Java + 少量库（轻量）、轻量框架。用户选择**纯 Java + OkHttp + Jackson**。

- **分析**：这是正确的选择。一个 harness 内核不需要 Spring Boot 的依赖注入和自动配置——那些恰恰是"框架替你做了"的部分，而项目的核心要求是"你自己实现"。

#### 决策 4：领域定位

用户将 harness 定位为「帮用户构建 AI 陪伴 Web 应用」，而非通用 coding agent。

- **分析**：这是一个关键的差异化决策。通用 coding agent（如 Claude Code）已有成熟产品，而「领域特化」让这个项目有独特价值。记忆系统在这个领域有天然落地场景——陪伴类应用的核心是"记住对话、建立关系"。

#### 决策 5：双层记忆

智能体提出 harness 记忆 + 生成项目记忆两层架构，用户同意并强调"生成项目的记忆额外重要"。

- **分析**：这催生了 ProjectMemoryRuntime 的设计——它不仅是 harness 内部的一个模块，更是作为运行时库注入到每个生成项目中的独立能力。这种"记忆即产品"的思路提升了项目的工程价值。

### 1.3 AI 提出的关键建议及我的处理

| AI 建议 | 采纳/修正 | 理由 |
|---------|----------|------|
| TypeScript 作为首选语言 | 推翻，选 Java | 用户偏好和熟悉度优先 |
| 治理护栏作为重点维度 | 推翻，选记忆 | 用户对记忆系统更有兴趣 |
| Spring Boot 架构 | 推翻，选纯 Java | harness 内核不需要框架层 |
| 三层架构（循环/记忆/生成） | 采纳 | 清晰的分层设计 |
| Mustache 模板引擎 | 采纳 | 轻量、无逻辑模板，适合代码生成 |
| SQLite 作为默认存储 | 采纳 | 嵌入式、零运维 |
| 三个机制演示（护栏/反馈/记忆） | 采纳 | 直接对应课程要求 |

### 1.4 智能体追问的好问题

1. **"六个维度中，你想把哪个作为主要贡献？"** — 这个问题迫使我在项目开始前就确定核心方向，避免六个维度都浅尝辄止。
2. **"harness 自身记忆和生成项目记忆，两层都做还是聚焦一层？"** — 这个问题直接催生了双层记忆架构，也是整个 SPEC 中最有价值的设计决策之一。
3. **"用户如何与 harness 交互？一次性指令还是多轮对话？"** — 这个问题影响 CLI 设计、AgentLoop 的停机判断、以及整个用户体验流程。
4. **"Java 生态方面，你偏好哪种架构风格？"** — 提供了四种选择并说明了各自的 trade-off。

### 1.5 Brainstorming 技能评价

**做得好的地方：**
- 严格执行"一次一个问题"，不让人感到 overwhelmed
- 问题有递进性：从技术栈 → 架构 → 领域 → 细节
- 多选题为主，降低决策成本
- 每个决策后确认才进入下一步

**可改进的地方：**
- 对「记忆」维度的追问深度不够——只问了一层还是两层，没有追问"什么样的记忆有价值"、"嵌入向量如何获取"等更具体的问题
- 没有追问"为什么是 AI 陪伴应用"——如果深挖，可能发现更精确的领域痛点
- "风险与未决问题"部分基本由我（AI）自行填写，没有引导用户思考"你觉得最大的风险是什么"

## 二、冷启动验证

### 2.1 验证方案说明

按照通用要求 §4.5，应在正式实现前用一个不同的智能体验证 SPEC 和 PLAN 的清晰度。由于项目实际流程中实现已经在进行，冷启动验证采取了以下替代方案：

- **实现 agent 类型**：Claude Code（Anthropic Claude）作为主开发智能体
- **冷启动验证策略**：在 Phase 3（记忆系统，重点维度）实现阶段，使用不同模型（Haiku vs Sonnet）派发子任务，观察它们对 SPEC/PLAN 的理解差异

### 2.2 发现的问题

| 问题 | 来源 | SPEC/PLAN 修订 |
|------|------|---------------|
| InMemoryStore 的 `listRecent()` 返回顺序不明确 | Haiku agent 实现为 oldest-first，但预期是 newest-first | 在 PLAN Task 18 的测试中明确了排序要求 |
| ProjectContext 被多个 agent 重复创建 | PLAN 没有明确标注哪些文件是"已创建，仅修改" | 后续 agent 的 prompt 中增加了已有文件清单 |
| Maven 未安装在系统 PATH 中 | Windows 环境差异 | agent 自行下载并配置了 Maven，后续 agent prompt 中增加 Maven 路径说明 |

### 2.3 对 SPEC/PLAN 的修订

1. **InMemoryStore 排序规范**：在 Task 18 测试中显式添加了 `get(0).key().isEqualTo("c")` 以强制 newest-first 排序
2. **agent prompt 模板优化**：后续 agent 的 prompt 中增加了 | 已有文件 | 不可重建 | 说明，避免重复创建
3. **Maven 路径标准化**：在 prompt 中提供 Maven 安装路径指导

## 三、反思

### 3.1 SPEC 质量如何影响实现质量

整体而言，SPEC 中描述越精确的模块，subagent 实现越准确。例如：

- **LLM 抽象层**（接口定义非常明确）→ 3 个 agent 都在 1 次尝试内完成
- **记忆系统**（概念清晰但细节留给实现者）→ 出现了排序方向不一致的问题，需要在测试中纠正

"规约不清导致 subagent 偏离"的典型案例是 InMemoryStore.listRecent() 的排序方向——SPEC 只说"返回最近 N 条"，但没有说按什么顺序。Haiku agent 返回了 oldest-first，而 Sonnet agent 预期 newest-first。这说明即使是一个"最近"这样的日常词汇，也需要在 SPEC 中明确定义。

### 3.2 最有效的 prompt 策略

1. **精确代码模板**：直接给出关键类的完整代码，agent 只需要转录+测试，几乎不出错
2. **接口先行**：先让 agent 创建接口文件，后续 agent 面向接口实现，减少耦合错误
3. **测试即规约**：在 prompt 中给出完整的测试代码，agent 通过测试驱动实现
